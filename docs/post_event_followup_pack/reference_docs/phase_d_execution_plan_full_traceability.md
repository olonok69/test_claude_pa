# Phase D Execution Plan (Alternative): No Suppression + Full Neo4j Traceability

## Assumptions
- No suppression is applied between personal agendas and engagement campaigns.
- Both campaigns are written to the same Neo4j database.
- Full recommendation traceability is available (run-level metadata on recommendations and campaign deliveries).
- Engagement campaign includes its own randomized control group.
- Post-show attendance is available via post-analysis relationships/files.

## Analysis objective
Run a **multi-campaign performance analysis** in Phase D with:
- causal estimates for both personal agendas and engagement,
- cross-campaign interaction effects using shared visitor history.

## Required traceability model (minimum)
For every recommendation and delivery event, preserve:
- `run_id`
- `run_mode` (`personal_agendas`, `engagement`, `post_analysis`)
- `campaign_id`
- `generated_at`
- `delivered_at` (or delivery status)
- `visitor_id` and session/exhibitor target id
- `control_group` flag (when applicable)
- `control_group_type` (`personal_agendas` / `engagement`)
- `randomization_seed` and `allocation_version`
- `source_step/version` (pipeline provenance)

For engagement specifically, also preserve:
- `engagement_control_eligible` (0/1)
- `engagement_control_assigned` (0/1)
- stratification keys used during sampling (e.g. new/returning, sector)

## Phase D analysis steps

### 1) Build a run ledger
Create a run-level table from Neo4j metadata:
- one row per campaign run,
- counts of targeted, delivered, withheld, failed,
- date windows and content scope.

Output: `run_ledger_phase_d`.

### 2) Build visitor-campaign exposure timeline
For each visitor, order exposures over time:
- campaign sequence (PA run 1 -> engagement -> PA incremental),
- delivered vs withheld flags,
- recommendation set IDs.

Output: `visitor_exposure_timeline`.

### 3) Attach post-show outcomes
Join timeline with attendance outcomes:
- session attendance (`assisted_session_this_year`),
- show attendance (`registered_to_show`),
- timing windows (pre/post delivery).

Output: `visitor_outcome_table`.

### 4) Personal agendas causal block (primary)
Use only PA runs with control assignment:
- treatment hit rate,
- control organic hit rate,
- absolute and relative lift,
- two-proportion z-test + 95% CI,
- stratified results by new/returning and major audience segments.

This remains the causal gold-standard section.

### 5) Engagement causal block (new)
Use engagement treatment/control cohorts generated by randomized assignment:
- engagement treatment hit rate,
- engagement control organic hit rate,
- absolute and relative lift,
- two-proportion z-test + 95% CI,
- stratified lift (new/returning, sector, function) with minimum-N thresholds.

### 6) Engagement performance block (operational + descriptive)
In addition to causal metrics, compute:
- delivery coverage,
- recommendation-attendance alignment,
- attendance among engaged visitors,
- overlap with PA-exposed visitors,
- sequence effects (engagement-only vs PA+engagement visitors).

### 7) Cross-campaign interaction analysis
Using timeline data, quantify whether multi-touch exposure changes outcomes:
- compare cohorts: `PA only`, `Engagement only`, `PA + Engagement`.
- optionally fit regression/propensity-adjusted models (exploratory).

Label this section as model-based/exposure-sequence inference (complements randomized blocks).

### 8) Attribution framing
Use a layered attribution approach:
- Layer A: randomized causal (PA)
- Layer B: randomized causal (Engagement)
- Layer C: sequence/interaction modeling (exploratory)

### 9) QA and reconciliation gates
Hard checks before publishing:
- run counts in reports match Neo4j counts by `run_id`,
- no duplicate recommendation events per visitor/session/run,
- exposure-to-outcome join rate above threshold,
- control split integrity for PA runs.
- control split integrity for engagement runs.
- randomization balance checks (e.g. new/returning, sector) for both campaign control groups.

### 10) Reporting structure
Publish one report with three sections:
1. **Causal Lift (Personal Agendas)**
2. **Causal Lift (Engagement)**
3. **Cross-Campaign Interaction Effects + Operational Diagnostics**

Each chart/table must include a `causal_status` tag:
- `causal_randomized`
- `descriptive_noncausal`
- `model_based_exploratory`

## Practical KPIs for this alternative
- PA causal lift (overall + segment)
- Engagement causal lift (overall + segment)
- Engagement recommendation hit alignment
- Multi-touch uplift proxy (`PA+ENG` vs `PA only`)
- Coverage and delivery quality by run
- Stability of results across run dates

## Why this plan is stronger than suppression-only for Phase D
- Preserves full visitor journey and recommendation lineage.
- Enables richer cross-campaign diagnostics.
- Reduces ambiguity in what recommendation came from which run.
- Makes audits/reconciliation substantially easier.

## Limitations
- Multi-touch modeling can suggest influence, but randomized evidence remains strongest for PA runs.

## Engagement control implementation notes
- Start with 5–10% engagement control allocation, then tune using power analysis.
- Prefer stratified randomization (at minimum by new/returning) to reduce cohort skew.
- Freeze allocation per `run_id` to avoid reassignment drift.
- Export separate engagement control files for delivery tooling and QA.

## Deliverables
- `run_ledger_phase_d`
- `visitor_exposure_timeline`
- `visitor_outcome_table`
- final Phase D report with causal-status labels
- reconciliation appendix by run_id
